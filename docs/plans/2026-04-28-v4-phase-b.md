## 1. Goal

Build the first ★ bundled skill (`plan-pack`) end-to-end as a tracer-bullet spike, then incrementally extend it from PRD-only to the full 4-doc generator (PRD + ARCH + ADR + UI_GUIDE). This phase establishes the templates, infrastructure, and patterns that the remaining 9 standard bundles will copy.

## 2. Why this phase, why this approach

Three Phase B candidates were on the table: `plan-pack ★`, `debugger ★`, and cross-cutting `SCOPE.md`. We chose `plan-pack` because:

- It is the V4 identity centerpiece — it must exist for the "self-sufficient library" promise to hold
- Once it lands, the parallel-dispatch / harness-prepend / orchestrator-only patterns are *demonstrated* and become copy-pasteable for the rest of the bundle lineup
- Failure mode is contained: PRD-only ships first as a working artifact, so the next Phase B-N is opt-in extension, not a recovery

We reject the "full ambition" approach (4 docs in one spike) in favor of **tracer-bullet → mechanism-by-mechanism → 4-doc serial extension**. Reasoning:

- Phase A proved that small commits + dogfood gates compose well
- Doing 4 docs simultaneously hides which mechanism actually broke when dogfood fails
- Each spike merge keeps `plan-pack` shippable for the user — extension is an upgrade, not a rewrite

## 3. Phase structure

```
Phase B (V4 plan-pack ★ spike)
├── B-1: PRD spike            (~1 week)   — sub-spikes 1–7 in full
├── B-2: ARCH spike           (2–3 days)  — sub-spikes 4–7 reused with new template
├── B-3: ADR spike            (2–3 days)  — sub-spikes 4–7 reused with new template
├── B-4: UI_GUIDE spike       (3–4 days)  — sub-spikes 4–7 reused + AI-slop antipattern table baked into template
└── B-5: 4-doc integration    (1–2 days)  — true 4-way parallel dispatch, cross-doc consistency, blank-Mac simulation
```

Phase B passes when **all five sub-phase dogfood gates** pass on master and each phase's trace is committed under `docs/dogfood/`.

### 3.1 Why ARCH/ADR/UI_GUIDE only repeat sub-spikes 4–7

Sub-spikes 1–3 (`run_dir.py`, SKILL skeleton, `harness.py` prepend helper) are infrastructure. They are written once in B-1 and reused unchanged in B-2 through B-4. This is enforced as gate item B2.4: `git diff B-1..B-2 -- server/run_dir.py server/harness.py` must be empty.

### 3.2 Where parallel dispatch is verified

| Phase | Parallel-dispatch event | Purpose |
|---|---|---|
| B-1 | PRD body + AC bash split into 2 sub-tasks → single message, 2 Agent calls | Mechanism check (small) |
| B-5 | All 4 docs regenerated in iteration 2 → single message, 4 Agent calls | Real 4-way verification |

B-2 through B-4 use single dispatch only; the parallel path is exercised in B-1 and B-5 alone. This avoids inflating the middle phases.

## 4. plan-pack bundle skeleton (Phase B-1 PRD-only state)

### 4.1 Directory layout

```
~/.claude/skills/assemble/bundled/plan-pack/
├── SKILL.md
└── templates/
    └── PRD.md.template
```

In B-2 through B-4, `templates/` gains `ARCHITECTURE.md.template`, `ADR.md.template`, and `UI_GUIDE.md.template`. The SKILL.md grows new sections each phase but never replaces existing ones.

### 4.2 SKILL.md frontmatter and preamble

```markdown
---
name: plan-pack
description: Plan stage ★ bundle — produce PRD/ARCH/ADR/UI_GUIDE with iteration. (Phase B-1: PRD only)
---

[HARNESS RULES — 무시 금지]
1. 불확실하면 추측 금지, 사용자 질문 우선
2. 과설계 금지, YAGNI
3. 요청 범위 밖 코드 임의 수정 금지
4. 버그 수정 시 재현 테스트 → 실패 확인 → 수정 → 재검증 루프
```

The four-line preamble is duplicated in `_shared/harness-preamble.md` (loaded by `server/harness.py` and prepended to every dispatched prompt at runtime). The SKILL.md copy is a backup for the rare case where the main Claude does light IO directly.

### 4.3 Sub-agent role mapping (Phase B-1)

| Step | Work | Role | Preferred | Fallback |
|---|---|---|---|---|
| 1 | User interview (8 questions) | (main, AskUserQuestion) | — | — |
| 2 | PRD body draft | `plan-implementation` | `Plan` | `general-purpose` |
| 3 | Acceptance Criteria bash draft | `plan-implementation` | `Plan` | `general-purpose` |
| 4 | Consistency review | `second-opinion` | `codex:codex-rescue`, `superpowers:code-reviewer` | `general-purpose` |
| 5 | Combine + write `<run_dir>/PRD.md` | (main, write_run_artifact) | — | — |

Steps 2 and 3 fire as a single message with two Agent calls (parallel-dispatch verification location *a*).

### 4.4 Workflow (B-1 state)

```
0. Resolve run_dir. If PRD.md exists → iteration mode (load existing as input)
1. Main: AskUserQuestion 8 prompts (single batched call)
2,3. Parallel dispatch — PRD body + AC bash, harness preamble auto-prepended
4. Consistency dispatch (second-opinion)
5. Merge + write runs/<rid>/PRD.md
6. AskUserQuestion: "next iteration?" → yes returns to step 1 (with accumulated answers); no ends
```

### 4.5 Eight interview questions (B-1)

1. What are you building? (one sentence)
2. Who uses it? (1–3 user types)
3. Three core features?
4. **Three things explicitly excluded from MVP?** (harness #2 enforcement)
5. One-line success criterion?
6. One AC bash command — how do you externally verify "it works"?
7. One-line design direction? (seed for UI_GUIDE later)
8. One risk or open question?

## 5. Phase B-1 sub-spike breakdown

Each sub-spike = one commit. Verification passes before the next starts.

### Sub-spike 1: run_dir infrastructure
- **Build:** `server/run_dir.py` as a *new* module (not `state_store.py` extension — `state_store` owns `progress.json`, `run_dir` owns artifact files; separation keeps each concern testable). `write_run_artifact(rid, filename, content) -> path` helper. Auto-creates the directory; safe permissions; uses the same `update_json_locked`-style file lock pattern where two concurrent writes to the same artifact are possible.
- **Tests:** 2–3 unit tests (happy path, concurrent write, missing directory).
- **Verify:** `pytest` green; calling helper actually creates `runs/<rid>/PRD.md`.
- **Depends on:** none.

### Sub-spike 2: plan-pack SKILL.md skeleton
- **Build:** `bundled/plan-pack/SKILL.md` + `bundled/plan-pack/templates/PRD.md.template` per §4.
- **Verify:** `inventory.scan()` recognizes plan-pack with `bundled=true`; ★-prefixed in plan stage menu; 500-char `body_excerpt` truncates cleanly.
- **Depends on:** Sub-spike 1 (output path referenced inside SKILL.md).

### Sub-spike 3: harness preamble prepend mechanism
- **Build:** `server/harness.py` with `wrap_with_preamble(prompt) -> str`. Reads `_shared/harness-preamble.md` once (cached). Wraps as `[HARNESS RULES] ... [TASK] {prompt}`.
- **Tests:** unit tests for correct prepending, empty-prompt handling, missing-preamble fallback.
- **Verify:** `pytest` green; SKILL.md preamble text matches the file (single source of truth).
- **Depends on:** none. Could parallel with 1–2 in theory, but commits stay serial for review clarity.

### Sub-spike 4: single dispatch for PRD body
- **Build:** SKILL workflow runs (a) the 8-question AskUserQuestion, (b) one `Plan`-agent dispatch wrapped via `wrap_with_preamble`, (c) writes PRD.md. AC bash, consistency review, and iteration are not yet wired.
- **Verify:** one dogfood run produces a PRD.md with body filled and AC bash empty; the dispatched prompt log shows the harness preamble prepended.
- **Depends on:** Sub-spikes 1, 2, 3.

### Sub-spike 5: parallel dispatch (PRD body + AC bash)
- **Build:** Workflow extends to *single message, 2 Agent calls*. Body and AC bash dispatch in parallel; results merged into PRD.md.
- **Verify:** trace shows two Agent calls in one turn; PRD.md contains at least one AC bash line that is actually executable (curl, grep, etc.).
- **Depends on:** Sub-spike 4.

### Sub-spike 6: consistency review (second-opinion)
- **Build:** Workflow step 4 — dispatch the drafted PRD to a `second-opinion` role with a *challenge* prompt. Result either appended as a `## Review notes` section or absorbed by main and rewritten into the body.
- **Verify:** dispatched output contains at least one rebuttal/flaw/correction (not bare agreement); fallback context arrives in the prompt when preferred agents are absent.
- **Depends on:** Sub-spike 5.

### Sub-spike 7: iteration round-trip
- **Build:** Workflow ends with `AskUserQuestion: "next iteration?"`. On yes: collect more user answers, re-run steps 2–3 with the existing PRD.md as input, write the update. Spike covers exactly one iteration; counts of 3–7 are a later track.
- **Verify:** the resulting PRD.md diff is meaningful (not boilerplate restatement); user's new answers visibly absorbed in the trace.
- **Depends on:** Sub-spike 6.

### Phase B-1 dogfood gate (final)

Run `/assemble <real but small task>` end-to-end. Pick plan-pack at the plan stage, generate PRD.md, run one iteration, exit. Then:
- [ ] harness preamble prepended in dispatched-prompt trace
- [ ] at least one single-message ≥2 Agent-calls trace
- [ ] `runs/<rid>/PRD.md` exists with ≥1 AC bash and ≥1 MVP-excluded item
- [ ] PRD diff after iteration 1 is non-trivial
- [ ] zero harness-rule violations on review

Pass → merge to master → start Phase B-2.

## 6. Phase B-2 / B-3 / B-4 (repeating extension)

Each phase = five steps following the same shape.

### B-2: ARCH (2–3 days)

| Step | Work | Verification |
|---|---|---|
| 1 | Add `templates/ARCHITECTURE.md.template` (stack, directory tree, patterns, data flow) | template format applied |
| 2 | Extend SKILL.md with ARCH section, role mapping, workflow; add 5–8 ARCH interview questions | inventory rescan OK |
| 3 | Dogfood: after PRD, single dispatch → `runs/<rid>/ARCHITECTURE.md` | one trace, file exists |
| 4 | second-opinion now spans both docs (PRD↔ARCH consistency) | one cross-check trace |
| 5 | Confirm iteration step also touches ARCH | one iteration trace |

AC bash is PRD-specific and intentionally not added to ARCH. ARCH's verifiable artifact is the directory tree / data-flow diagram.

### B-3: ADR (2–3 days)

Same five-step shape. ADR.md.template = *N decisions + reasoning + tradeoffs*. Interview asks for three core decisions, rejected alternatives, and tradeoffs. Review checks for missing rationale and missing tradeoffs.

### B-4: UI_GUIDE (3–4 days)

Same five-step shape. UI_GUIDE.md.template includes the **shipping-default AI-slop antipattern table** verbatim from the V4 spec memory:

- gradient-text, glass morphism, backdrop-blur, all-purple palettes
- meaningless emoji bombing
- residual "Lorem ipsum" / "placeholder"
- unresolved "TODO: fill in"
- ad-copy clichés like "innovative" / "next-generation"

Review cross-checks the antipattern list against the design direction in PRD §6. Self-check: UI_GUIDE.md must not itself contain antipattern artifacts (gate B4.3).

## 8. Dogfood gates (full table)

### Common (every phase)

| # | Item | Method | Fail condition |
|---|---|---|---|
| C1 | run_dir artifact present | `test -f runs/<rid>/<doc>.md` | file missing |
| C2 | harness preamble prepended | grep `HARNESS RULES` in dispatched-prompt log | zero hits |
| C3 | orchestrator-only adherence | trace review for main-turn Edit/Write outside the artifact write | main writes body directly |
| C4 | zero harness-rule violations | manual trace review (guessing, over-engineering, scope creep, skipped verification) | ≥1 violation |
| C5 | roles.json fallback healthy | run with preferred agent absent | dispatch fails instead of fallback firing |

### Phase-specific gates

**B-1**
- B1.1 PRD.md has ≥1 AC bash block (`grep -E '^\`\`\`bash' runs/<rid>/PRD.md \| wc -l` ≥ 1)
- B1.2 ≥1 MVP-excluded item
- B1.3 single-message 2-Agent-calls trace exists
- B1.4 iteration 1 PRD diff ≥5 lines
- B1.5 second-opinion produced ≥1 flaw/rebuttal (not bare agreement)

**B-2**
- B2.1 ARCHITECTURE.md exists
- B2.2 directory tree and data-flow sections each ≥5 lines
- B2.3 ≥1 PRD↔ARCH cross-flaw detected
- B2.4 `git diff B-1..B-2 -- server/run_dir.py server/harness.py` empty (infra reuse)

**B-3**
- B3.1 ADR.md exists
- B3.2 ≥3 decisions, each with explicit tradeoff or rejected-alternative subsection
- B3.3 zero "TBD" / "TODO" / "미정" tokens

**B-4**
- B4.1 UI_GUIDE.md exists with antipattern table (≥6 items)
- B4.2 antipattern cross-check trace recorded against PRD design direction
- B4.3 UI_GUIDE body itself contains zero antipattern instances (no gradient-text, glass, "innovative/next-gen" copy)

**B-5 (V4 release gate)**
- B5.1 single turn with 4 simultaneous Agent calls
- B5.2 each of 4 docs has ≥5-line diff after iteration 2
- B5.3 ≥2 cross-doc flaws detected by second-opinion
- B5.4 blank-Mac simulation (`ASSEMBLE_BUNDLED_ONLY=1`, see Phase B-5 step 4a) completes plan stage in ≤5 min with plan-pack only
- B5.5 dispatch trace under preferred-absent environment (constructed by temporarily renaming the relevant agent definitions in `~/.claude/agents/` for the test run, then restoring) shows fallback context populated and dispatch firing against `general-purpose`

### Operating procedure

- Each sub-spike commit lists its gate items in the message (Phase A pattern)
- Each phase ends with `docs/dogfood/phase-b-<n>.md` capturing trace excerpts and the gate result table, committed to master
- Failure → revert that sub-spike, analyse root cause, retry. Failure pattern is logged in memory (seed for cross-cutting C learning capture)

## 10. Hard prohibitions (V4 identity)

- Main Claude must not write PRD body directly (orchestrator-only violation)
- SKILL.md must not hard-code home-Mac-specific agent names; role names only
- harness_framework code must not be copy-pasted; pattern/idea borrow only
- No "mandatory gate" forcing the user to finish plan; every step is skippable
- Iteration count must not be force-multiplied — user "plan complete" exits immediately
